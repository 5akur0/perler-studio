# Responsive UI Audit — 2026-07-07

Full-surface audit of Perler Studio across phones, tablets, and desktops.
Method: Playwright walked every screen and every bead-flow phase at 18 viewport
configurations (254 captured states), with programmatic checks (horizontal
overflow, touch-target size, input font size, accessible names, text clipping)
plus visual inspection of ~40 representative screenshots. Branch:
`feat/choose-library-reshape`, dev server, default theme.

## Viewport matrix

| Class | Sizes tested | Emulation |
|---|---|---|
| Phone portrait | 320×568, 360×640, 375×667, 390×844, 430×932 | touch, DPR 2 |
| Phone landscape | 844×390, 932×430 | touch, DPR 2 |
| Tablet | 768×1024 (portrait), 1024×768, 1194×834, 1366×1024 | touch, DPR 2 |
| Desktop | 800×900, 1100×800, 1280×620, 1280×720, 1440×900, 1920×1080, 2560×1440 | mouse, DPR 1 |

States per viewport: home; bead choose / place / inspect / iron / cool / finish;
settings, share, confirm, draw-import modals; drawing studio (+ phone palette
sheet); gallery (+ submit modal); collection; community (both tabs).

## Anti-patterns verdict

**Pass.** No gradient text, no colored side-stripes, no eyebrow kickers, no
identical-card slop, no emoji icons, hover states move only color/shadow. The
craft-desk scenes (sieve, sticky note, hanging needle, iron) read as authored,
not generated. Minor blemishes only: the share modal's export grid leaves an
orphan cell, and the destructive 「清空作品集」 is the most prominent element on
an almost-empty collection screen.

## Executive summary

- Health score **17/20 (Good)**. Zero P0s: every flow is completable at every
  tested size, including 320×568 and a 620px-tall desktop window.
- Issues found: **3 × P1, 4 × P2, 8 × P3**.
- Top issues:
  1. On **every phone size**, the finish screen's last action (分享小红书) hangs
     6px below the viewport with its container scrolled shut — the share entry
     point at the exact "出片" moment looks broken.
  2. Community 留言板 inputs are 15px → iOS zooms on focus (violates the
     project's 16px-Input Rule; `input-font-regression` doesn't cover them).
  3. At ≤375px phones the drawing studio hides 图片转图纸 / 导入图纸 / 清空 in a
     horizontally scrollable strip with **no visible affordance** at rest.
  4. Pro-Max-class landscape phones (932×430 ≥ 861px) bypass the "rotate to
     portrait" overlay and get the tablet layout, contradicting DESIGN.md's
     orientation policy.

## Findings by severity

### P1 — fix before release

**[P1-1] Finish action stack overflows the viewport by 6px on all phones**
- **Location**: mobile finish phase; `#stageControls` finish rows inside
  `.bead-studio-grid` flex column (`src/styles/responsive.css` mobile-working
  block).
- **Category**: Responsive.
- **Evidence**: at 320×568, 390×844, and 430×932 the 分享小红书 button's bottom
  edge = viewport + 6px, with `#studioGrid` holding a 22px internal scroll. The
  constant 6px across all heights indicates a fixed accounting error (bottom
  padding / safe-area / gap not subtracted from the canvas flex), not content
  pressure.
- **Impact**: the share CTA at the celebration moment renders clipped; the
  container scrolls but nothing signals it. Sharing is a core success metric
  (PRODUCT.md #2), so this is the worst possible screen to look broken on.
- **Recommendation**: make the finish control stack report its true height so
  the canvas flex shrinks by the missing ~28px, or trim one gap tier in the
  finish rows. Add a regression: in mobile finish phase, last stage-control
  bottom ≤ `window.innerHeight`.
- **Suggested command**: `/impeccable adapt`

**[P1-2] Community inputs at 15px trigger iOS focus zoom**
- **Location**: `src/styles/components.css:3013` — `.community-nickname,
  .community-textarea { font-size: var(--fs-md); }` (15px).
- **Category**: Accessibility / own-rule violation (DESIGN.md The 16px-Input
  Rule).
- **Impact**: iPhone Safari zooms the page on focus; the flow's calm rhythm
  breaks exactly when a player tries to leave a message.
- **Recommendation**: `font-size: var(--fs-input, 16px)`; extend
  `scripts/input-font-regression.mjs` to cover dynamically rendered community
  fields (it currently misses them).
- **Suggested command**: `/impeccable harden`

**[P1-3] Drawing studio: hidden actions with no affordance on small phones**
- **Location**: mobile drawing footer strip (`.drawing-footer-actions` +
  mobile CSS ordering), states `12-draw-studio` at 320–375px.
- **Category**: Responsive / discoverability.
- **Evidence**: visible at rest: 开始拼豆 / 导出图纸 / 投稿. Off-screen right at
  x=316+ (viewport 320): 图片转图纸, 导入图纸, 清空. At 320 nothing peeks past
  the edge; at 390 a sliver of 图片转图纸 shows. The strip scrolls, but a
  first-time phone user has no signal that 图片转图纸 — a headline feature —
  exists.
- **Impact**: feature discovery on the smallest phones is close to zero.
- **Recommendation**: guarantee a half-button peek at all widths (e.g. clamp
  item widths so the 4th item always crosses the edge), or add a fade/chevron
  edge affordance. Note the strip's CSS `order` also diverges from DOM order,
  so Tab first lands on an off-screen button (see P3-7).
- **Suggested command**: `/impeccable adapt`

### P2 — fix in the next pass

**[P2-1] ≥861px landscape phones bypass the rotate overlay**
- **Location**: `src/styles/responsive.css:1784` — overlay fires at
  `(hover:none) and (pointer:coarse) and (max-width:860px) and (orientation:landscape)`.
- **Category**: Responsive / policy consistency (DESIGN.md: 横屏手机由
  `.orientation-overlay` 提示旋转).
- **Evidence**: 844×390 correctly shows 「请将手机竖屏使用」; 932×430 (iPhone
  Pro Max class, many large Androids ≥ 861 CSS px in landscape) instead renders
  the tablet three-column layout at 430px height.
- **Impact**: an unplanned layout variant on popular hardware; usable in
  testing, but with sub-44px workflow steps and cards clipped mid-row.
- **Recommendation**: decide explicitly — either raise the overlay cutoff (e.g.
  `max-width: 950px` + short-height guard like `max-height: 500px`), or bless
  this layout and fix its touch targets.
- **Suggested command**: `/impeccable adapt`

**[P2-2] Workflow-progress steps are 56×34px on touch tablets**
- **Location**: topbar `.workflow-step` in the 861–1179 compact band
  (`src/styles/responsive.css`), measured on 1024×768 and 932×430 touch.
- **Category**: Accessibility (project floor `--tap-min: 44px`; WCAG 2.5.8 AA
  24px passes, own bar doesn't).
- **Impact**: the six phase buttons are the primary navigation of the bead flow
  on iPad.
- **Recommendation**: expand hit area via the same `::before` inset trick the
  library card buttons use — no visual change needed.
- **Suggested command**: `/impeccable harden`

**[P2-3] Active workflow step is visually weaker than done steps**
- **Location**: `.workflow-step.active` vs `.workflow-step.done` (all
  platforms, all phases; e.g. place phase at 1440×900).
- **Category**: Visual hierarchy.
- **Evidence**: done steps get solid brand badges + ink labels; the active step
  gets a pale tint badge. Scanning "where am I" reads the last done step (e.g.
  选图) as current while in 摆放.
- **Recommendation**: invert emphasis — the active step should carry the
  strongest treatment (solid badge or ring + `--brand-ink` label); done steps
  can drop to a check glyph.
- **Suggested command**: `/impeccable polish`

**[P2-4] Community screen has no content column on desktop**
- **Location**: `.community-layout` / `.community-pane` at ≥1280 widths.
- **Category**: Layout.
- **Evidence**: at 1440×900 the nickname input and textarea run ~1330px wide;
  the two tabs stretch to 700px each. Every other screen respects a shell or
  card width.
- **Recommendation**: cap the pane (e.g. `max-width: 720px; margin-inline:
  auto`) consistent with `.studio-shell` conventions.
- **Suggested command**: `/impeccable layout`

### P3 — polish when convenient

- **[P3-1] Library rename target is 24px tall.** `.library-card-name`
  (components.css:1073) opts out of the 44px floor without the `::before`
  expansion its star/delete siblings have. Same fix as P2-2.
- **[P3-2] Palette search input is 36px tall** (`#drawPaletteSearch`), below
  `--field-h` 42 / `--tap-min` 44, on all platforms.
- **[P3-3] 320px-only near-misses**: draw tool buttons 41px wide; settings
  theme swatches 42px wide. Fine at ≥360.
- **[P3-4] Board doesn't scale on large desktops.** The pegboard renders ~the
  same absolute size at 1440×900 and 2560×1440, leaving a vast empty desk
  (and at 1100×800 the board is smaller than the sticky note beside it). If
  bead-size realism is intentional, fill the desk with ambient props;
  otherwise raise the board's size cap with viewport area.
- **[P3-5] Destructive 清空作品集 is the collection screen's loudest element**,
  even with one saved work. Consider a quiet danger-text treatment (the bead
  studio already has a `danger-text-button` precedent).
- **[P3-6] Share modal export grid orphan cell** — 导出竖图/导出方图/纯作品图
  leave an empty fourth slot; make 纯作品图 span the row or reflow to one
  column.
- **[P3-7] Draw footer strip: visual order ≠ DOM order.** CSS `order` places
  开始拼豆 first visually while 图片转图纸 is first in DOM, so keyboard focus
  enters the strip off-screen.
- **[P3-8] Bottom-center toasts cover controls on phones** (bean-box chips in
  place, 保存作品 in finish) for their full 2000–2600ms. Consider nudging the
  mobile toast above the action stack.

## Patterns & systemic issues

1. **The 44px floor erodes exactly where `min-height: 0` opt-outs appear.**
   Star/delete solved it with `::before` insets; rename, workflow steps, and
   the palette search didn't get the same treatment. One shared
   `.tap-target::before` utility would close the class of bugs.
2. **Dynamically rendered fields dodge the regression suite.** The 16px input
   rule and 44px rule are both enforced by scripts, but community fields
   (rendered by `community.js`) and phase-dependent controls aren't in their
   sweep. Point the regressions at live DOM states, not just static HTML.
3. **Off-screen content relies on undiscoverable inner scroll on phones**
   (finish stack, draw strip, and — benignly — the choose grid). The choose
   grid gets away with it because a card row always peeks; apply the same
   "always peek" rule to every mobile scroll container.

## Positive findings

- **Zero horizontal overflow in all 254 states** — remarkable across this many
  breakpoints; the token-driven `@media` redefinition strategy works.
- **Every icon button has an accessible name**; not one unnamed control was
  found across the whole app. Modals trap focus, restore it, and close on Esc;
  the canvas has a real keyboard placement path (verified end-to-end: the
  audit placed beads with arrows + space).
- **Orientation gates behave exactly as designed** at standard sizes: 844×390
  → 竖屏 prompt, 768×1024 → 横屏 prompt, with `aria-live` on the overlay.
- **Platform-true layouts, not shrunken clones**: phone gets the stacked
  board-on-desk with only used colors; iPad gets a lighter three-column rig
  with tight board framing; desktop gets the full sieve/needle/note craft desk
  and a 检查台 with magnifier + fusion preview. This is PRODUCT.md's platform
  strategy actually shipped.
- **Loading and empty states are designed**: gallery skeleton tiles → empty
  state with its own CTA; community empty copy in brand voice; sandbox grade
  labeled honestly as 评级 沙盒 through finish, share, and collection.
- **Narrow desktop windows (800×900, mouse) degrade gracefully** into the
  stacked layout; short desktops (1280×620) keep every control reachable via
  panel scroll.
- Both font roles load and are used per the Clear-vs-Cute rule (numbers and
  codes stay in Noto; titles/CTAs in LXGW).

## Coverage notes

Not covered by this audit: the 5-theme × screen matrix (default 雾青 only, plus
the settings previews), real-device Safari/iOS behavior, reduced-motion
verification, contrast re-measurement (last measured 2026-06-13 per DESIGN.md),
and runtime performance profiling. The gallery rendered its empty state — the
production gallery had no public patterns at audit time, so populated-grid
layout wasn't exercised.
